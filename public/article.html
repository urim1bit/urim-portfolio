Migrating from XML to Jetpack Compose

Oct 12, 2024 • 5 min read

For years, Android UI was built using XML layouts. It worked, but as applications grew more complex, managing the state between XML views and Kotlin code became a massive headache. Enter Jetpack Compose.

In this article, I want to share my personal experience migrating a legacy application entirely to Compose, the challenges I faced, and why I believe it's the future of Android development.

The Problem with XML

In the traditional View system, UI is stateful. When data changes, you have to manually update the UI by finding the view (using findViewById or View Binding) and setting its new properties. This imperative approach often leads to bugs where the UI and the data get out of sync.

"Compose changes the paradigm from imperative UI to declarative UI. You describe what the UI should look like for a given state, and the framework handles the rest."

A Simple Comparison

Let's look at a simple example. Here is how we used to define a basic text element in XML:

<TextView
    android:id="@+id/titleText"
    android:layout_width="wrap_content"
    android:layout_height="wrap_content"
    android:text="Hello World"
    android:textSize="20sp" />

And here is the exact same element using Jetpack Compose in Kotlin:

@Composable
fun TitleText() {
    Text(
        text = "Hello World",
        fontSize = 20.sp
    )
}

The Migration Strategy

You don't have to rewrite your whole app at once. Compose is highly interoperable. You can add a ComposeView inside an existing XML layout, or you can host Android Views inside a Composable using AndroidView.

My strategy was bottom-up: I started by migrating the smallest, standalone UI components (like buttons and list items) and eventually worked my way up to entire screens and navigation graphs.

Conclusion

Migrating to Jetpack Compose has drastically reduced my boilerplate code and made UI development actually fun again. If you haven't started learning it yet, now is the time.
